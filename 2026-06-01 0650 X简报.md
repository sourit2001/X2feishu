### 一、God of Prompt：温度参数的“葬礼”，真正的恐慌点你还没看到

**主要内容：**  
这位 Prompt 工程大神发出了一个重磅信号：所有前沿 AI 提供商都“杀死”了温度参数（temperature）。具体来说：
- OpenAI 在 o1、o3 和 GPT-5 上移除了温度调节；
- Google 警告不要在 Gemini 3 上更改温度；
- Anthropic 在 Claude Opus 4.7 中弃用了它，并在 4.8 延续这一做法。

但 God of Prompt 认为，大部分人恐慌错了方向。温度参数的消失并非意味着我们失去了控制随机性的能力，而是整个范式正在转变——**模型已经足够智能，不再需要用户手动微调“创造力”与“确定性”的杠杆**。同时，他顺手安利了自己的 Claude 技能包（Claude skills bundle），并转发了另外两条动态：一条是对比 Seedance 2.0 和 Gemini Omni Flash 的评论（强调两者对比本身是伪命题），另一条则展示了如何仅凭一个 Prompt 就抓取 OpenAI 一整周的 X 分析数据，无需 API 或代码。

**核心观点：**  
- 温度参数死亡 = 开发者必须从“调参数”转向“调上下文/提示词”。
- 真正值得关注的不是“失去了什么调整能力”，而是“模型理解人类意图的能力已进化到不再需要这个旋钮”。
- 未来 Prompt 工程的竞争点，在于如何引导模型在“无温度”环境中做出你想要的输出。

### 二、Ethan Mollick：别迷恋全自动 Agent，AI 应该学会“什么时候提问”

**主要内容：**  
沃顿商学院教授 Ethan Mollick 对当前流行的 `/goal` 等完全自动化 AI Agent 模式泼了一盆冷水。他认为，这些 Agent 虽然很酷，但**并不能作为未来人机协作的良好模型**。相反，我们真正需要的 Agent 是：知道何时向用户提出**好问题**。例如，在遇到瓶颈时主动求助、在涉及人类审美偏好时主动确认、或者只因为它觉得某个细节你会感兴趣。

**核心观点：**  
- 全自动 vs. 协作式：自动化 Agent 追求“少干预”，协作式 Agent 追求“恰到好处的干预”。
- 好问题的标准：让用户觉得 **“幸好你问了”** ，而不是“又被打断了”。
- 这项工作机制比堆砌能力更重要：未来的 Agent 评测指标应该包括“提问质量”。

**实用工具/资源：**  
- 无特定工具推荐，但理念可应用于设计 Agent 的“停顿-询问”逻辑。

- 当前 Agent 透明性不足：缺乏中间进度条、里程碑标记或“过早放弃”警告。
- 人类对 Agent 的信任阈值正在被重新定义：16小时是放弃与坚持的临界点。

## 🔥 综合启发：对未来 AI 开发者与使用者的 3 条行动建议

1. **拥抱“无温度”时代，重构你的 Prompt 体系**  
   温度参数消失是好事——它逼着我们从“调旋钮”转向“精炼指令”。学习新的控制技巧，比如使用 `top_p`、`frequency_penalty`、以及更细粒度的系统提示示例，来替代温度的功能。God of Prompt 的 Claude 技能包也许是个起点。

2. **设计 Agent 时，嵌入“提问机制”比追求全自动更重要**  
   Ethan Mollick 的洞察非常关键：未来的 Agent 不是“做完所有事”，而是“在正确时间点把人类拉进来”。你的 Agent 必须有内在的“不确定性感知”模块——当置信度低、涉及主观判断、或耗时超过预期时，主动抛出高质量问题。这比盲目跑16小时更有价值。

3. **给 Agent 加上“防空转护盾”：里程碑、暂停与 kill 决策**  
   Vasu 的困境揭示了当前 Agent 平台的一大缺口：缺乏运行时可视化。如果你在开发复杂任务 Agent，务必提供：
   - 中间成果快照
   - 进度百分比（即使近似）
   - 自动暂停阈值（如超过 N 小时无新进展）
   - 用户可自定义的“继续/放弃/修改目标”按钮  
   这样，用户就不会在“期待”与“怀疑”之间煎熬。

最后，记住一条元建议：**AI 工具越强大，人类对“何时介入”的判断越重要。** 温度已死，但提问永生。下次跑 Agent 前，先问自己：它遇到问题时，会来找我吗？
